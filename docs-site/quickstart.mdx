Vendo setup is one command and one paste. `npx vendo init` scans your app and
scaffolds the entire server side; you paste the `<VendoRoot>` wrap it prints,
drop in the overlay, and send a turn.

<Note>
  Vendo's built-in agent is in beta. Most teams should bring their own agent:
  keep the loop you already run and spread Vendo's tool pack into it. Jump
  straight to the [AI SDK](/existing-agents/ai-sdk) or
  [Mastra](/existing-agents/mastra) quickstart. This page wires Vendo as the
  whole agent, loop and UI included.
</Note>

## Install and init

```bash
npm install @vendoai/vendo
npx vendo init
```

Init asks nothing on the happy path. It detects your framework and auth from
`package.json`, then writes everything Vendo needs:

- the catch-all route `app/api/vendo/[...vendo]/route.ts` holding the whole
  `createVendo` composition
- an empty component registry, `vendo/registry.tsx`, for the branded
  components the agent may render
- your API extracted into `.vendo/tools.json`, and any `"use server"` actions
  registered as agent tools through a generated `vendo-actions.ts` map
- your brand extracted into `.vendo/theme.json`
- a model key: if no provider key is set, init offers a metered Vendo Cloud
  starter key inline

It never edits code you wrote. The full inventory and how detection decides:
[vendo init](/connect/vendo-init).

The generated route is yours to read and change, but there is nothing left to
write. The scaffold looks like this:

```ts
// app/api/vendo/[...vendo]/route.ts (generated by `vendo init`)
import { authJs } from "@vendoai/vendo/auth/auth-js";
import { createVendo, nextVendoHandler } from "@vendoai/vendo/server";
import { serverActions } from "./vendo-actions";
import { registry } from "@/vendo/registry";

const vendo = createVendo({
  auth: authJs(),    // detected from package.json
  catalog: registry, // the components the agent may render
  serverActions,     // generated when "use server" actions are detected
  policy: {},        // .vendo/policy.json: destructive asks, reads run
});

export const { GET, POST, PUT, PATCH, DELETE } = nextVendoHandler(vendo);
```

Every key is optional, and there is no `model` line on purpose: a bare
`createVendo()` resolves a model from the environment and defaults to
anonymous ephemeral sessions and PGlite persistence. Every key lives in
[Handler options](/reference/handler-options).

The handler takes a standard `Request` and returns a standard `Response`, so
it runs in Next.js, Express, Fastify, Hono, Bun, or any Node-compatible
runtime (Node 20 or later); non-Next wiring is
[below](#wire-the-express-adapter).

## Paste the root

The one line init never writes is the client wrap. It prints these exact
lines for you to paste:

```tsx
// app/layout.tsx
import { VendoRoot } from "@vendoai/vendo/react";
import { registry } from "@/vendo/registry";
import theme from "@/.vendo/theme.json";
import type { VendoTheme } from "@vendoai/vendo";

// then wrap the app:
<VendoRoot components={registry} theme={theme as VendoTheme}>
  {children}
</VendoRoot>
```

The `theme` prop applies the brand init captured; the cast narrows
TypeScript's widened JSON-module string literals.

## Add the overlay

`VendoOverlay` from `@vendoai/ui/chrome` is the ready-made agent surface: a
brand-styled launcher pill plus a dialog panel over your product.

```tsx
import { VendoOverlay } from "@vendoai/ui/chrome";

export function Assistant() {
  return <VendoOverlay />;
}
```

Move the launcher with `launcher="bottom-left"`, or set `launcher="none"` and
drive it yourself with the `useVendoOverlay` hook; see
[Overlay control](/reference/hooks#overlay-control-usevendooverlay).

## Send the first turn

Start your dev server, open the overlay, and ask for a view, such as "Show my
overdue invoices by customer." Then gate on doctor:

```bash
npx vendo doctor
```

Doctor checks the wiring, makes one live `/status` round trip, and runs live
HTTP probes that confirm present-mode credentials forward and that `actAs`
(if configured) mints material your host accepts. See the
[CLI reference](/reference/cli#vendo-doctor-dir) for probe details.

One requirement before you ship: production must set `VENDO_BASE_URL` to the
app's public origin, or a tool call that needs credentials fails loud instead
of running unauthenticated. Init writes a `http://localhost:3000` default
into `.env.example`; see
[Environment variables](/reference/environment-variables#host-process).

## Wire the Express adapter

On Express, `vendo init` proposes a diff-gated `vendo/server.ts` holding the
same `createVendo` composition plus a Node HTTP-to-fetch adapter, and two
pastes remain instead of one. Mount the adapter under one base path, and wrap
the client root:

```ts
import express from "express";
import { mountVendo } from "./vendo/server.js";

const app = express();
app.use("/api/vendo", mountVendo());
app.listen(3000);
```

```tsx
import { VendoRoot } from "@vendoai/vendo/react";
import { registry } from "./vendo/registry";
import theme from "./.vendo/theme.json";

root.render(
  <VendoRoot components={registry} theme={theme}>
    <App />
  </VendoRoot>,
);
```

`vendo doctor` reports broken until both pastes are done.

### Other runtimes

On Fastify, Hono, or Bun, decline init's proposed diffs and mount
`vendo.handler` under one base path yourself; the Node adapter init generates
for Express is the pattern to copy. An adapter must:

- Pass every method (GET, POST, PUT, PATCH, DELETE) through unchanged.
- Preserve streaming for `POST /threads`, never buffering the response body.
- Reconstruct the request URL from `req.originalUrl`, not `req.url`, when mounted
  with `app.use` on a base path (Express strips the mount path from `req.url`).
- Keep cookie and authorization headers available to `principal(request)` and to
  present-mode same-origin tool calls.
- Return multi-value `Set-Cookie` as a header array, not a joined string.
