This page takes you from install to a first working agent turn in your own
app, on Next.js or Express and other Node runtimes. You wire one server
composition and one client root; `vendo init` scaffolds the rest.

<Note>
  Vendo's built-in agent is in beta. Most teams should
  [bring their own agent](/existing-agents) and use Vendo's tools and surfaces
  from their existing loop. This quickstart wires Vendo as the whole agent,
  loop and UI included.
</Note>

## Install

```bash
npm install @vendoai/vendo
npx vendo init
```

Vendo's handler takes a standard `Request` and returns a standard `Response`,
so it runs in Next.js, Express, Fastify, Hono, Bun, or any Node-compatible
runtime (Node 20 or later). Init flags live in the
[CLI reference](/reference/cli).

    ## The two files you own

    `vendo init` asks nothing on the happy path. It writes an empty
    `vendo/registry.tsx` and the catch-all route
    `app/api/vendo/[...vendo]/route.ts` wired to it, extracts your tools and
    theme into `.vendo/`, resolves a model key, and prints the one
    `<VendoRoot components={registry}>` line that is yours to paste. It never
    edits code you wrote. How the pieces fit: see
    [How Vendo works](/concepts/architecture).

    ### `vendo/registry.tsx`

    See [Host components](/connect/host-components) for the full registration
    reference.

    ```tsx
    import type { ComponentRegistry } from "@vendoai/core";
    import { z } from "zod";
    import { SpendingDonut } from "@/components/charts/spending-donut";

    export const registry = {
      SpendingDonut: {
        component: SpendingDonut,
        description: "Spending by category. Use for where-did-my-money-go requests.",
        props: z.object({
          slices: z.array(z.object({ category: z.string(), amount: z.number() })),
        }),
        examples: ['{"slices":[{"category":"dining","amount":342.18}]}'],
      },
    } satisfies ComponentRegistry;
    ```

    ### The catch-all route

    ```ts
    // app/api/vendo/[...vendo]/route.ts (what `vendo init` scaffolds)
    import { anthropic } from "@ai-sdk/anthropic";
    import { authJs } from "@vendoai/vendo/auth/auth-js";
    import { createVendo, nextVendoHandler } from "@vendoai/vendo/server";
    import { registry } from "@/vendo/registry";

    const vendo = createVendo({
      model: anthropic("claude-sonnet-4-6"),
      auth: authJs(),
      catalog: registry,
      policy: {}, // .vendo/policy.json: destructive asks, reads run
    });

    export const { GET, POST, PUT, PATCH, DELETE } = nextVendoHandler(vendo);
    ```

    Every key is optional. `authJs()` reads `AUTH_SECRET` and derives the
    principal from the session-token claims; presets also exist for Clerk,
    Supabase, Auth0, and a generic JWT scheme
    ([actAs presets](/connect/act-as-presets)). Every `createVendo` key lives
    in [Handler options](/reference/handler-options).

    ## Add the root

    ```tsx
    import { VendoRoot } from "@vendoai/vendo/react";
    import { registry } from "@/vendo/registry";

    export function Root({ children }: { children: React.ReactNode }) {
      return <VendoRoot components={registry}>{children}</VendoRoot>;
    }
    ```

    ## Add the overlay

    `VendoOverlay` from `@vendoai/ui/chrome` is the ready-made agent surface:
    a brand-styled launcher pill plus a dialog panel over your product.

    ```tsx
    import { VendoOverlay } from "@vendoai/ui/chrome";

    export function Assistant() {
      return <VendoOverlay />;
    }
    ```

    Move the launcher with `launcher="bottom-left"`, or set `launcher="none"`
    when your host triggers the overlay itself. For programmatic control,
    pair it with the `useVendoOverlay` hook; see
    [Overlay control](/reference/hooks#overlay-control-usevendooverlay).

    ## Wire the Express adapter

    On Express, `vendo init` proposes a diff-gated `vendo/server.ts` holding a
    `createVendo()` composition and a Node HTTP-to-fetch adapter, plus a
    starter `vendo/ai.ts` model module. Two wiring steps remain. Mount the
    exported adapter under one base path, normally `/api/vendo`:

    ```ts
    import express from "express";
    import { mountVendo } from "./vendo/server.js";

    const app = express();
    app.use("/api/vendo", mountVendo());
    app.listen(3000);
    ```

    Wrap the client root in `<VendoRoot>`:

    ```tsx
    import { VendoRoot } from "@vendoai/vendo/react";

    root.render(
      <VendoRoot>
        <App />
      </VendoRoot>,
    );
    ```

    ## Composition

    The generated `vendo/server.ts` calls `createVendo` once during boot:

    ```ts
    import { createVendo } from "@vendoai/vendo/server";
    import { model } from "./ai";

    const vendo = createVendo({
      model,
      principal: async () => null,
      policy: {}, // .vendo/policy.json: destructive asks, reads run
    });
    ```

    Replace `principal` with your session resolver. It receives the `Request` and
    returns a `Principal`, or `null` for an anonymous session:

    ```ts
    principal: async (request) => {
      const user = await resolveSession(request);
      return user ? { kind: "user", subject: user.id } : null;
    },
    ```

    ## Mount the handler directly

    On any other runtime (Fastify, Hono, Bun), decline init's proposed diffs
    and mount `vendo.handler` under one base path yourself; the Node adapter
    init generates for Express is the pattern to copy. An adapter must:

    - Pass every method (GET, POST, PUT, PATCH, DELETE) through unchanged.
    - Preserve streaming for `POST /threads`, never buffering the response body.
    - Reconstruct the request URL from `req.originalUrl`, not `req.url`, when mounted
      with `app.use` on a base path (Express strips the mount path from `req.url`).
    - Keep cookie and authorization headers available to `principal(request)` and to
      present-mode same-origin tool calls.
    - Return multi-value `Set-Cookie` as a header array, not a joined string.

    Credential forwarding also needs the `VENDO_BASE_URL` requirement under
    [Set the trusted origin](#set-the-trusted-origin) below.

## Send the first turn

Open your conversation surface and ask for a view, such as "Show my overdue
invoices by customer." The client sends `POST /threads` and reads an AI SDK UI
message stream. App views arrive as `data-vendo-view` parts. Approval metadata
arrives as `data-vendo-approval` parts.

The scaffolded `policy: {}` reads `.vendo/policy.json` (destructive tools ask,
reads run); every policy shape lives in
[Handler options](/reference/handler-options#composition-configuration).

## Set the trusted origin

Present tool calls forward the inbound `cookie` and `authorization` headers
only when the wire trusts its own origin. Development learns that origin from
the first request; every other environment keeps it untrusted, so a spoofed
`Host` header can never exfiltrate credentials. Production deployments must
set `VENDO_BASE_URL` to the app's public origin:

```bash
VENDO_BASE_URL=https://app.example.com
```

Without it, a present-mode tool call that needs credentials fails loud
instead of running unauthenticated, and `vendo doctor` reports the missing
variable. `vendo init` writes a `http://localhost:3000` default into
`.env.example`. `VENDO_BASE_URL` does more than gate present-mode
credentials; see
[Environment variables](/reference/environment-variables#host-process).

## Persistence

A bare `createVendo()` defaults to an env-resolved model, anonymous ephemeral
sessions, and PGlite persistence. See
[Vendo Cloud](/deploy/vendo-cloud#hosted-store) for the store slot under
`VENDO_API_KEY`.

## Serve your product to MCP clients

Outside agents (Claude, ChatGPT, Cursor, any MCP client) can connect through
the experimental MCP door: one flag, `mcp: true`. See [MCP door](/capabilities/mcp).

  <Tab title="Next.js">
    ```bash
    npx vendo doctor
    npx vendo sync
    ```

    Doctor checks wiring, probes `/status`, and runs live HTTP probes that
    confirm present-mode credentials forward and that `actAs` (if configured)
    mints material your host accepts. Sync extracts tools and remix baselines.
    See the [CLI reference](/reference/cli#vendo-doctor-dir) for probe details.
  </Tab>
  <Tab title="Express & other runtimes">
    Run `vendo doctor` against the running server:

    ```bash
    npx vendo doctor
    ```

    For an Express host it checks the `createVendo` composition and the
    `<VendoRoot>` mount, makes one live `/status` round trip, and runs the
    same credential and `actAs` probes as the Next.js path. It reports broken
    until both wiring steps are done. See the
    [CLI reference](/reference/cli#vendo-doctor-dir) for probe details.
  </Tab>
