Vendo Cloud is the hosted backend for cloud-gated surfaces: sharing frozen
app copies, publishing app versions, hosted sandboxes and store, and
org-scoped keys and members. Every one of those capabilities keeps a
bring-your-own path (your own Postgres, an E2B sandbox account, your own
model key), so local development never needs Cloud. Reach for it when you
want cross-host sharing, published copies, or a machine key for CI.

## Prerequisites

- A Vendo Cloud account at `https://console.vendo.run`.
- The `vendo` bin on your PATH (`npx vendo` also works).

## Provide an API key to the runtime

Cloud-gated runtime calls read `VENDO_API_KEY` from the host process. When it is
unset or empty, `vendo.apps.share()` and `vendo.apps.publish()` throw
`cloud-required` and the mapped HTTP status is 402. Set it wherever your handler
runs:

<Warning>
  Setting the key does more than unlock the gated surfaces: it also fills
  every adapter slot your composition left unset with the Cloud default, the
  [hosted sandbox](#hosted-sandboxes) and the [hosted store](#hosted-store).
  With no `store` passed, your Vendo data moves from local PGlite to Vendo
  Cloud. An explicitly passed adapter always wins; pass
  `store: createStore(...)` to keep persistence local.
</Warning>

```bash
export VENDO_API_KEY=vnd_live_...
```

## Sign in from the CLI

`vendo login` is the headline flow: it opens your browser at the approval
page (a non-TTY caller gets the URL and pairing code printed instead), you
approve the code while signed in to the console, and the minted
`VENDO_API_KEY` is written to `.env.local` — never printed.

```bash
vendo login
#   1. Open https://console.vendo.run/claim?code=BCDF-GHJK
#   2. Confirm the code: BCDF-GHJK
# Approved — wrote VENDO_API_KEY (…a1b2) to .env.local.
```

`vendo cloud device-login` is the same command, kept as an alias.

### Email-OTP fallback

`vendo cloud login <email>` sends a 6-10 digit code to your inbox, prompts for
it on stdin, and stores the returned session at `~/.vendo/cloud-session.json`
(file mode 0600). The exact code length depends on your Vendo Cloud project's
OTP configuration. The user commands reuse that session.

```bash
vendo cloud login you@example.com
# Enter the code sent to you@example.com: 12345678
# { "loggedIn": true, "mode": "email", "email": "you@example.com" }
```

If you already hold a Vendo Cloud access token, skip the OTP and store it
directly:

```bash
vendo cloud login --token <jwt>
```

Sign out with `vendo cloud logout`. Check that the session works with
`vendo cloud whoami`, which prints the organizations it can reach.

## Create and rotate keys

```bash
vendo cloud keys create --org <orgId> --name "ci-publisher"
vendo cloud keys list --org <orgId>
vendo cloud keys revoke --org <orgId> --id <keyId>
```

`--org` is optional only when your account has exactly one organization.

There is no validate endpoint and no local entitlements cache. `vendo doctor`
checks that `VENDO_API_KEY` is present and well formed locally; a bad or
revoked key surfaces on the first real service call with
`Invalid or revoked API key (401)`.

Every key-authed call carries sanitized deployment-identity headers
(`X-Vendo-Deployment-Host`, `X-Vendo-Deployment-Name`), so usage and
deployments appear in the console, `vendo cloud usage`, and
`vendo cloud deployments` from real traffic. To confirm a CI key works, make
a real call and check the exit code.

## Point at a different Vendo Cloud URL

By default the runtime and CLI both target `https://console.vendo.run`. Set
`VENDO_CLOUD_URL` to override, for example against a staging environment:

```bash
export VENDO_CLOUD_URL=https://console.staging.vendo.run
```

Individual CLI commands also accept `--api-url <url>` for a one-off override.

## Share and publish

With `VENDO_API_KEY` set, the composition's apps runtime calls the hosted API:

```ts
import { createVendo } from "@vendoai/vendo/server";

const vendo = createVendo({ /* model, principal, store, ... */ });

const snapshot = await vendo.apps.share(appId, ctx);
const record = await vendo.apps.publish(appId, ctx);
```

`share` returns a `ShareSnapshot` (a frozen copy: `id`, `doc`, `createdAt`).
`publish` returns a `PublishRecord` with the assigned `version` (`id`, `appId`,
`version`, `createdAt`). Both responses are validated against the frozen wire
schemas (`shareSnapshotSchema`, `publishRecordSchema`) before they resolve.

## Automations run in Cloud automatically

There is nothing to deploy. With `VENDO_API_KEY` set, the automations you
build sync to Vendo Cloud as you save them and run on Vendo's schedulers —
no separate push step, no CLI command.

External triggers (Stripe, Slack, and the rest of the Composio connector
catalog) ride the Composio connectors: connect the account once in the
console and the hosted trigger registers itself. Host-event automations
(`on: { kind: "host-event", ... }`) still fire in your own process, since
they react to events your process emits.

## Error handling

Cloud calls surface the shared error taxonomy (see
[Troubleshooting](/deploy/troubleshooting)):

- `cloud-required` (HTTP 402): no `VENDO_API_KEY`, the key is invalid, or the
  operation's meter is exhausted.
- `validation`, `not-found`, `conflict`, `blocked`: propagated from the hosted
  API with the same codes the wire uses.

Handle `cloud-required` by setting `VENDO_API_KEY` — `vendo login` claims one
into `.env.local` — or create a named key with `vendo cloud keys create`.

## Cloud reference

### Key format

API keys use the format `vnd_` followed by 40 hex characters. `vendo init
--cloud-key` rejects a malformed key before writing it; `vendo doctor` warns
on one already in the environment. Neither makes a network call to do so.

### The `fn:` limitation for synced automations

Automations whose steps use `fn:` tools (calls back into the host
application's process) warn on sync. Hosted sandboxes cannot reach your
machine in v1, so those steps will fail or park when the automation fires
in the cloud. Either replace them with tools the hosted runtime can
execute, or pass an explicit local `store:` (see the warning above) to keep
that composition's schedule- and external-trigger automations firing on
your own machine instead of Cloud's.

### Hosted sandboxes

`cloudSandbox` is the hosted sandbox adapter behind machine-backed apps
(layers 2 and 3 of [the machine model](/concepts/generated-ui#the-three-layers)).
Reach for it when you want app machines without an E2B account. You normally
never construct it: with `VENDO_API_KEY` set, `createVendo` fills the unset
sandbox slot with it automatically. An explicitly passed `sandbox:` adapter,
or `E2B_API_KEY` with the `e2b` package installed, always wins over that
default. Usage meters as `sandbox_minutes`.

To construct it explicitly (the adapter itself never reads the environment):

```ts
import { cloudSandbox, createVendo } from "@vendoai/vendo/server";

const vendo = createVendo({
  // model, principal, store, ...
  sandbox: cloudSandbox({ apiKey: process.env.VENDO_API_KEY! }),
});
```

`cloudSandbox(options)` accepts:

- `apiKey` (required): Vendo Cloud API key.
- `baseUrl`: console base URL override; `createVendo` passes
  `VENDO_CLOUD_URL` when it composes the default.
- `timeoutMs`: per-request abort timeout in milliseconds. Defaults to five
  minutes.

Machines boot from Cloud's own pooled base image (Node plus the in-box agent);
`VENDO_BOX_TEMPLATE` does not apply. Snapshot references are prefixed
`vendo:v2:` and resume only against the console. The egress allowlist is
enforced HTTP(S)-only: raw TCP is severed even to allowlisted hosts, so a
direct database connection from the box never works while the HTTPS store
callback does. Errors map to the shared taxonomy: invalid or revoked keys
(HTTP 401) and exhausted meters (HTTP 402) surface as `cloud-required`,
console outages as `sandbox-unavailable`; with no key and no adapter at all
the sandbox slot is simply unconfigured, so machine paths fail with
`sandbox-unavailable`. See [Server API](/reference/server-api) for the
`SandboxAdapter` seam and
[Generated UI and apps](/concepts/generated-ui) for the layer model.

### Hosted store

`hostedStore` is the Cloud persistence adapter: a plain `StoreAdapter`
speaking to the console's store routes, tenant resolved server-side from the
key's org. You normally never construct it: with `VENDO_API_KEY` set and no
`store` passed, `createVendo` fills the store slot with it automatically,
so threads, apps, records, grants, and audit then persist in Vendo Cloud
instead of local PGlite. An explicitly passed `store:` always wins over that
default:

```ts
import { createStore, createVendo } from "@vendoai/vendo/server";

const vendo = createVendo({
  // model, principal, ...
  store: createStore({ url: process.env.DATABASE_URL }), // data stays yours
});
```

To construct it explicitly (the adapter itself never reads the environment):

```ts
import { createVendo, hostedStore } from "@vendoai/vendo/server";

const vendo = createVendo({
  store: hostedStore({ apiKey: process.env.VENDO_API_KEY! }),
});
```

`hostedStore(options)` accepts `apiKey` (required), `baseUrl` (console base
URL override; `createVendo` passes `VENDO_CLOUD_URL` when it composes the
default), and `timeoutMs` (per-request abort timeout).

Stored secrets are excluded by construction: the hosted wire has no secrets
surface, and the `storeSecrets`/`secretStore` helpers keep requiring a local
store handle. They fail loudly against the hosted store rather than sending
secret values to the console. The composition's secrets default (`envSecrets`)
reads your process environment either way. `ensureSchema` is a client no-op
(the service owns its migrations), and `eraseStore` cascades server-side
exactly like the local implementation.

For the full CLI surface, see [CLI](/reference/cli).
