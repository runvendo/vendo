description: "How Vendo scopes data to users, persists anonymous sessions with TTL sweep, merges them on sign-in, and reserves the vendo: namespace."

Every wire request resolves a `Principal` through the `principal(req)` seam
you pass to `createVendo`. The returned `subject` scopes threads, apps,
approvals, grants, activity, and runs. This page covers the identity model
and the two behaviors that ride on it: **anonymous auto-merge** on first
sign-in, and **organization workspaces** (Vendo Cloud only).

## Principal kinds

`Principal.kind` is `"user"` or `"org"`. Your `principal(req)` resolver may
only return `kind: "user"`. Organization principals are minted server-side
by Vendo Cloud and are never resolved directly by the OSS host.

```ts
principal: async (request) => {
  const user = await resolveSession(request);
  return user ? { kind: "user", subject: user.id } : null;
},
```

Returning `null` gives the client an ephemeral session — see
[Anonymous sessions](#anonymous-sessions) below.

## The reserved `vendo:` namespace

Subjects that start with `vendo:` are reserved for Vendo-owned identities.
Two are defined today:

- `vendo:webhook:<source>` — webhook deliveries execute under this subject
  when no principal is otherwise attached to the request.
- `vendo:org:<orgId>` — reserved for Vendo Cloud organization workspaces.
  The OSS wire never mints this subject itself.

A `principal(req)` resolver that returns a `vendo:*` subject, or a
`kind: "org"` principal, is rejected at the wire with a `blocked` error.
Mint your own subjects under whatever prefix your product uses (for example
`user:`); leave the `vendo:` prefix alone.

If you have audit readers or dashboards that filter on webhook subjects,
update them to match `vendo:webhook:<source>` instead of the bare
`webhook:<source>` form.

## Anonymous sessions

When `principal(req)` returns `null`, Vendo mints an ephemeral session and
carries it in an opaque, `httpOnly` cookie — a random 128-bit session id
scoped to the wire base path. The cookie is a pointer, not a signed token:
the `vendo_sessions` row is the authority, so a forged or invented id
merely names an empty session.

Anonymous work is stored like any other subject — ordinary rows under the
anonymous subject — plus one `vendo_sessions` row recording the session's
last activity. Sessions survive process restarts and are shared across
instances, so no sticky routing is needed for anonymous traffic.

### Session lifecycle

Every request touches its session's `vendo_sessions` row. A session idle
past `sessions.ttlMs` (default 30 minutes) is swept: the row is deleted
along with every other row keyed to that anonymous subject — threads,
apps, per-app records and blobs, state, grants, approvals, runs.

The cookie itself stays valid. The next request from that client simply
gets a fresh, empty session under the same cookie. An app-scoped write
that races the sweep fails closed with `not-found` (`"session may have
expired"`) rather than recreating unreachable rows.

Sweeps run two ways so both long-lived and serverless hosts stay
consistent:

- **Background timer.** `createVendo` starts an unref'd interval that
  sweeps every `sessions.sweepIntervalMs` (default 60 s) and is torn down
  by `store.close()`. No wiring required.
- **Amortized on-request.** Every request runs a sweep at the top of the
  handler, before its own session is touched. A request that arrives past
  the TTL sweeps its own idle session first and then gets a fresh empty
  one. This keeps serverless platforms (no timer guarantee) in sync with
  long-lived hosts.

Tune the lifecycle when composing:

```ts
const vendo = createVendo({
  model,
  principal,
  sessions: {
    ttlMs: 30 * 60 * 1000,   // 30 min; 0 disables TTL sweep
    sweepIntervalMs: 60_000, // 60 s
  },
});
```

Setting `ttlMs: 0` disables the TTL sweep entirely; anonymous sessions
then persist until they are adopted on sign-in or erased explicitly.

### Auto-merge on sign-in

The first authenticated request that arrives with a valid anonymous cookie
**adopts** the session's data into the real user subject and then retires
the cookie (`Max-Age=0`). Users keep the conversation they started, the
apps they built, and the state those apps wrote. The merge is idempotent
and emits a `kind: "principal"` audit event (`detail.event: "anon-merge"`).

What is adopted:

- Threads (including the message the user was mid-typing when they signed in)
- Apps the anonymous session created, and their per-app records and blobs
- State written by those apps

What is **not** adopted (consent doesn't transfer identities):

- Grants and approvals — the anonymous session's authority does not carry
  over. The signed-in user re-approves on the next tool call.
- Connected accounts — external providers key credentials by subject, so
  users reconnect after signing in.
- Audit and run history — kept intact under their original subject.

Existing durable rows always win a collision. If the signed-in user
already owns a thread or app with the same id, the anonymous copy is
skipped rather than overwriting it. There is no configuration to turn
auto-merge off; it is the default behavior of the wire.

Nothing on the host side has to change to opt in: as long as
`principal(req)` returns `null` before the user signs in and a real
`Principal` afterward, Vendo does the rest.

## Organizations (Vendo Cloud)

Organization workspaces — shared apps, approvals, and grants under one org
subject — are a [Vendo Cloud](/deploy/vendo-cloud) capability, not an OSS
wire feature. The self-hosted wire always answers every `/api/vendo/orgs`
route, and any `?org=<id>` parameter on `/api/vendo/approvals` or
`/api/vendo/grants`, with a `cloud-required` posture error (HTTP 402) —
unconditionally, regardless of whether `VENDO_API_KEY` is set.
`GET /api/vendo/status` reports no `orgs` block.

Vendo Cloud manages its own accounts, members, and org-scoped keys
server-side. See `vendo cloud whoami`, `vendo cloud members`, and the
`--org` key scoping in the [CLI reference](/reference/cli) for the
Cloud-side org model.
