Keep your Mastra agent, its model, and your chat UI. Three steps add Vendo
to it: run `vendo init`, spread `vendoMastraTools` into the agent's tools,
and render tool outputs with one component.

## 1. Install and init

```bash
npm install @vendoai/vendo
npx vendo init
```

Init scaffolds the wire route (`app/api/vendo/[...vendo]/route.ts`), extracts
your host actions into agent tools, and captures your theme
([what init writes](/connect/vendo-init)). The wire route serves generated
apps and approvals to the embeds; your Mastra agent stays the loop.

Your agent needs the same `vendo` instance the wire route uses, so lift the
`createVendo` call init wrote into the generated route out to a shared
module both import. Copy it out of YOUR route verbatim — its keys mirror
what init found in your repo: an auth preset or a `principal` line,
`catalog`, and a `serverActions` map only when init detected `"use server"`
actions. Lifted, it looks like:

```ts
// src/lib/vendo.ts — the composition init wrote into the wire route,
// moved so your agent can import it too
import { createVendo } from "@vendoai/vendo/server";
import { registry } from "@/vendo/registry";

export const vendo = createVendo({
  // Who the wire's callers act as — init's line. Must resolve the same
  // subject your chat route sets in step 2.
  principal: async () => ({ kind: "user" as const, subject: "demo-user" }),
  catalog: registry, // the component registry init generated
  policy: {},        // .vendo/policy.json: destructive asks, reads run
});
```

```ts
// src/app/api/vendo/[...vendo]/route.ts — now just the handler
import { nextVendoHandler } from "@vendoai/vendo/server";
import { vendo } from "@/lib/vendo";

export const { GET, POST, PUT, PATCH, DELETE } = nextVendoHandler(vendo);
```

The route needs no `runtime` or `dynamic` exports: Node is the default
runtime, route handlers are dynamic by default, and on a Next 16 app with
`cacheComponents: true` an `export const dynamic` line is a build error.

## 2. Spread the pack

The agent definition gains the pack next to its own tools.
`vendoMastraTools` returns a Promise, which is why the agent uses Mastra's
tools-as-function form:

```ts
// src/mastra/agents/weather-agent.ts
import { Agent } from '@mastra/core/agent';
import { vendoMastraTools } from '@vendoai/vendo/mastra';
import { vendo } from '../../lib/vendo';
import { weatherTool } from '../tools/weather-tool';

export const weatherAgent = new Agent({
  id: 'weather-agent',
  name: 'Weather Agent',
  instructions: `…your instructions, plus a short note about the vendo_* tools…`,
  model: 'openai/gpt-4.1-mini',
  tools: async () => ({ weatherTool, ...(await vendoMastraTools(vendo)) }),
});
```

A Mastra agent definition is static (one definition serves every user), so
the shim takes no principal. Instead, set the caller's principal server-side
on the `RequestContext` in your chat route; a `vendo_*` call without one
fails closed.

<Note>
  Apps and approvals are principal-scoped: the principal you set here and
  the one your shared composition resolves (step 1) must name the same
  caller — resolve both from the same session (or the same shared constant,
  as the [full example](#the-full-example) does). If they disagree,
  everything the agent creates is invisible to the embeds, and the app
  embed polls "pending" forever.
</Note>

```ts
// src/app/api/chat/route.ts — Mastra's own Next.js guide route, two additions
import { handleChatStream } from '@mastra/ai-sdk';
import { RequestContext } from '@mastra/core/request-context';
import { createUIMessageStreamResponse } from 'ai';
import { mastra } from '@/mastra';
import { VENDO_PRINCIPAL_KEY } from '@vendoai/vendo/mastra';
import { vendo } from '@/lib/vendo';

export async function POST(req: Request) {
  const params = await req.json();

  await vendo.store.ensureSchema(); // lets guarded tools run before any wire request
  const requestContext = new RequestContext();
  // Resolve the caller from your own session — never trust the client. This
  // demo subject matches the one init writes into the wire route's principal.
  requestContext.set(VENDO_PRINCIPAL_KEY, { kind: 'user', subject: 'demo-user' });

  const stream = await handleChatStream({
    version: 'v6',
    mastra,
    agentId: 'weather-agent',
    params: { ...params, requestContext },
  });
  return createUIMessageStreamResponse({ stream });
}
```

<Note>
  Multi-turn tool use with GPT-5 reasoning models currently fails on history
  replay — pin `openai/gpt-4.1-mini`
  ([mastra-ai/mastra#9005](https://github.com/mastra-ai/mastra/issues/9005)).
</Note>

## 3. Render the embeds

Wrap the chat once in `VendoProvider` and hand finished tool outputs to
`<VendoToolResult>`: it renders the live app embed for a generated app, the
approval card for a parked call, and nothing for plain data. Mastra streams
tool calls as `dynamic-tool` or `tool-*` parts, so match both:

```tsx
// src/app/page.tsx — inside your existing message-parts loop
if (part.type === "dynamic-tool" || part.type.startsWith("tool-")) {
  const tool = part as ToolLikePart;
  return tool.state === "output-available"
    ? <VendoToolResult output={tool.output} />
    : <span>Running {tool.toolName ?? part.type.slice("tool-".length)}…</span>;
}
```

The frontend is identical to the [AI SDK quickstart](/existing-agents/ai-sdk):
the embed contract is framework-agnostic.

## Try it

<Warning>
  Two models, two credentials. Your loop keeps its own key (here
  `OPENAI_API_KEY`) — no Vendo key replaces it. Vendo's internal turns
  (app builds, the delegate) resolve theirs through the
  [dev-mode credential ladder](/connect/dev-mode), where an explicit
  provider key beats `VENDO_API_KEY` — so builds ride your OpenAI key
  (noticeably slower for builds) unless you set
  `VENDO_DEV_CREDENTIAL=vendo-cloud`. No key at all? `npx vendo login`
  mints a free metered dev key into `.env.local`. The done-criterion is a
  real chat turn that renders a Vendo tool output — run ask 2 yourself;
  doctor cannot check your loop's model.
</Warning>

One thread, three asks:

1. **"What's the weather in Paris?"** Your agent's own tool, untouched.
2. **"Make me a dashboard comparing weather in Paris, Tokyo and NYC."**
   `vendo_create_app` returns an app ref immediately and the app builds live
   inline in the chat.
3. **A guarded call from your own catalog.** Open `.vendo/tools.json`, pick
   a tool init labeled `destructive` (or one marked critical in
   `.vendo/overrides.json`), and ask for it in plain language. The call
   parks on the generated policy (destructive asks, reads run) and renders
   as an approval card; Approve executes it in place, Deny resolves it to
   "declined". (The [full example](#the-full-example) parks
   `vendo_send_trip_report` on "Email the trip report to ops@example.com" —
   but a stock starter has no email action, so ask for one that actually
   exists in YOUR extracted list.)

lines, every Vendo line inside `--- vendo:` comment fences), with a keyless

the envelope contract, the two-model split) live in
