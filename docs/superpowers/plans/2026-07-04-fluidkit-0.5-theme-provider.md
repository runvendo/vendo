> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship `fluidkit@0.5.0` with a semantic theme provider (`FluidThemeProvider`), per-component derivation, material/expressiveness knobs, React 19 test coverage, a reduced-motion contract test suite, and a playground brand demo — non-breaking for existing consumers.

### Task 1: Theme module — tokens, context, resolver

The heart of the release. Create the theme module in `src/theme/` (new directory), exported from the package root:

- The `FluidTheme` token type: color/identity tokens (accent, surface, text, mutedText, background, fontFamily, radius, mode) plus liquid-character tokens (material: glass/flat — caustics excluded from theming; intensity: the existing `LiquidIntensity` scale, reusing that vocabulary). All tokens optional.
- `FluidThemeProvider` — a plain React context provider carrying the theme. Nesting merges (inner overrides outer).
- The per-component resolver: given a component's identity and its explicit props, produce that component's resolved styling. Priority: explicit prop → theme derivation → the component's current built-in default. Critically, **only explicitly-set theme tokens derive anything** — an absent token never overrides a component default, so deliberate divergences (LiquidTabs defaults flat, LiquidToast near-solid) survive a colors-only theme. Derivation rules live in one table in this module so tuning the look is a one-file edit.
- The resolved shape is wider than `SurfaceStyleProps` where a component supports it: text-rendering components additionally resolve text color / mutedText / fontFamily; shape-bearing components resolve radius. The derivation table records, per component, which tokens apply; tokens with no consumer are simply inert for that component.

- [ ] Write failing unit tests first: no provider = identical resolution to today's defaults; provider tokens flow through; explicit props always win; nested providers merge; mode/material/expressiveness each change derivations as documented.
- [ ] Implement the module; run the test suite; commit.

### Task 2: Wire the resolver into main's components

- [ ] First, audit `origin/main`'s exports and classify them (Codex found the naive inventory wrong — e.g. Caustics, FlowStagger, GlassPanes, MeshGradient do NOT extend SurfaceStyleProps): (a) surface components speaking `SurfaceStyleProps` → full derivation; (b) text/shape components → typography/radius tokens where supported; (c) ambient/backdrop components (shader art) → color tokens only if they take colors, else untouched. Record the classification in the derivation table.
- [ ] For each themed component: add a themed-rendering test (component under a provider picks up derived tint/material), verify the existing untouched-default tests still pass, wire the resolver, commit per component or per small group.
- [ ] Full suite green; snapshot/visual sanity in the playground.

### Task 3: Degradation contract suite

Stage B retires Flowlet's static fallbacks, so degradation becomes fluidkit's tested promise. The contract is per-component, not one blunt rule — existing components intentionally allow opacity-only reduced-motion fades.

- [ ] One reduced-motion contract test per exported component: content renders and stays interactive; no looping animation or rAF-driven geometry; opacity-only fades allowed. Where a component already complies, the test pins it; where it doesn't, fix it in this task.
- [ ] Missing-capability fallback tests: no WebGL, no backdrop-filter, no refraction support → each degrades to its documented fallback rendering (assert the fallback, not just absence of crash).
- [ ] Document the contract in the README (a short "Motion accessibility & degradation" section).

### Task 4: React 19 alignment

Peers allow `>=18`, but nothing proves 19: dev deps are React 18 and CI only matrices Node versions.

- [ ] Add a React-major test matrix: dev environment stays on one major, and a CI leg installs `react@19`/`react-dom@19` (+ matching @types) via npm overrides before running the suite. Both legs run typecheck, tests, build, and the pack check.
- [ ] Fix anything the 19 leg surfaces; commit.

### Task 5: Playground brand demo

- [ ] Add a playground page ("Themes") mounting the same component set under three contrasting themes: a chromatic light brand, a monochrome/graphite brand, and a dark brand — with a material toggle (glass ↔ flat) and an expressiveness slider. This is the visual test bed and the doc-by-example for the provider.
- [ ] Screenshot the three themes for the PR.
