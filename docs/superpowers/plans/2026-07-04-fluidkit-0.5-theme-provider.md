# fluidkit 0.5 Theme Provider Implementation Plan (Stage A)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship `fluidkit@0.5.0` with a semantic theme provider (`FluidThemeProvider`), per-component derivation, material/expressiveness knobs, React 19 test coverage, a reduced-motion contract test suite, and a playground brand demo — non-breaking for existing consumers.

**Architecture:** A new theme module holds the token type, the provider, and a per-component resolver; each existing component swaps its inline prop defaults for the resolver (explicit props → theme-derived values → today's built-in defaults, in that priority). No component API changes.

### Task 1: Theme module — tokens, context, resolver

The heart of the release. Create the theme module in `src/theme/` (new directory), exported from the package root:

- The `FluidTheme` token type: color/identity tokens (accent, surface, text, mutedText, background, fontFamily, radius, mode) plus liquid-character tokens (material: glass/flat, expressiveness: the existing intensity scale). All tokens optional; sensible defaults documented.
- `FluidThemeProvider` — a plain React context provider carrying the theme. Nesting merges (inner overrides outer).
- The per-component resolver: given a component's identity and its explicit props, produce resolved surface props. Priority: explicit prop → theme derivation → the component's current built-in default. Derivation rules live in one table in this module (e.g. panels tint from accent at low alpha, dialogs slightly stronger, flat material fills from surface, dark mode adjusts alphas) so tuning the look is a one-file edit.

- [ ] Write failing unit tests first: no provider = identical resolution to today's defaults; provider tokens flow through; explicit props always win; nested providers merge; mode/material/expressiveness each change derivations as documented.
- [ ] Implement the module; run the test suite; commit.

### Task 2: Wire the resolver into main's components

Swap inline destructured defaults for the resolver in every exported surface component (Caustics, Droplets, FlowStagger, GlassPanes, LiquidButton, LiquidCard, LiquidDialog, LiquidPanel, LiquidText, LiquidTooltip, MeniscusDivider, MeshGradient, MorphSurface, Ripple, Silk, LiquidTabs, Thinking, VoiceBall — those that speak SurfaceStyleProps).

- [ ] For each component: add a themed-rendering test (component under a provider picks up derived tint/material), verify the existing untouched-default tests still pass, wire the resolver, commit per component or per small group.
- [ ] Full suite green; snapshot/visual sanity in the playground.

### Task 3: Reduced-motion contract suite

Stage B retires Flowlet's static fallbacks, so degradation becomes fluidkit's tested promise.

- [ ] One contract test per exported component: under `prefers-reduced-motion`, the component renders its content without entrance/looping animation (assert on the component's documented reduced-motion behavior; where a component already handles it, the test pins it; where one doesn't, fix it as part of this task).
- [ ] Document the contract in the README (a short "Motion accessibility" section).

### Task 4: React 19 alignment

- [ ] Add React 19 to the test matrix (either bump dev dependencies to 19 and keep 18 in a CI matrix leg, or the reverse — pick whichever the existing CI layout makes cheaper) so both majors run the full suite.
