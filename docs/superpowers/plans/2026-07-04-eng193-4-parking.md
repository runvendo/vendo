**Goal:** Ship-order item 4 from `docs/superpowers/specs/2026-07-02-eng193-permissions-design.md` §10: automations parking — **park the action, not the run** (§3 Moment 8, §4.6, §9 resolved decision 1). Built on the already-landed engine (item 1: tiers/`grantPolicy`/audit), consent channel + cards (item 2), and judge + breakers (item 3). Scope: a `ParkedAction` record + store CRUD, interpreter changes so an ungranted/critical tool inside a `for_each` or an agent step parks instead of failing the run, late standalone execution of an approved parked action, a dedicated consent-channel extension, list/resolve endpoints on both hosts, and a "Waiting on you" shell surface. No Trust screen, no fades, no diary, no steering — those are items 5–6.

**Architecture:**
- `packages/flowlet-runtime/src/automations/store.ts` (`AutomationEngineStore`, the ENG-188 engine seam — NOT the frozen `@flowlet/core` Store seam, mirroring where `AutomationGrant`/`PendingApproval` already live) gains a `ParkedAction` type + `StepStatus` gains `"parked"` + `AutomationRun` gains an additive `parkedCount: number` + four CRUD methods (`createParkedAction`/`listParkedActions`/`getParkedAction`/`resolveParkedAction`), principal-scoped, store-assigned ids, implemented on `InMemoryAutomationStore`.
- `packages/flowlet-runtime/src/automations/interpreter.ts` stays pure (no I/O — its own docstring's rule, unchanged): it accumulates `ParkedActionDraft[]` on its exec context exactly like it already accumulates `StepRecord[]`, and every `InterpretOutcome` variant carries `parkedActions: ParkedActionDraft[]`. Two call sites change: (a) `executeToolStep`'s `insideLoop` branch, which today throws `StepFailure` for an ungranted/critical gated tool, now records a draft, marks the step `"parked"`, and continues; (b) `gateAgentTool` (used by both loop-nested and top-level agent steps, and agentic mode), which today returns an `approval_required` error result, now ALSO records a draft alongside that same error result (the model still sees a clean "no" and routes around it — ruling 2(b)'s copy change).
- `packages/flowlet-runtime/src/automations/runner.ts`'s `AutomationRunner` persists every draft the interpreter returns (on every outcome branch — succeeded, failed, AND waiting_approval, since a run can park several actions before later pausing at a direct-step checkpoint) and increments the run's `parkedCount`. It gains a new method, `resolveParkedAction(scope, actionId, decision)`, sibling to the already-existing `resume()` — reusing the SAME per-automation `enqueue()` serialization `resume()` already relies on for its own "two rapid approvals must not double-execute" guarantee (spec §8's parked-critical-executes-exactly-once invariant rides the same mechanism, not a new one).
- Consent wire (`@flowlet/core/src/consent.ts`): `consentRequestSchema` — verified unused by any runtime call site today (grep: only referenced as a doc/type import) — becomes a `z.discriminatedUnion("kind", [...])` adding `"parked-action"` (additive, zero blast radius). A **separate**, purpose-built `parkedActionResolutionSchema` (`{actionId, decision: "yes"|"no"}`) backs the actual resolve wire call — see deviation #1 for why this does NOT reuse `consentResponseSchema`/the existing `/consent` route.
- `@flowlet/next` and the accounting demo each get a thin `parked-actions` route pair (list, resolve) mirroring the existing `consent.ts`/`consent-handler.ts` "thin adapter over a testable function" pattern exactly.
- `@flowlet/shell` gets a new, optional, standalone `WaitingList` component (not wired into `MessageList`/`ApprovalCard` — parked actions aren't thread items, they have no message/turn) plus a `useParkedActions` polling hook and a `parkedActions` seam on `ShellContextValue` (same optional-seam shape as `sendConsent`/`runQuery`). Hosts mount it wherever makes sense for them (see deviation #4).

## Plan deviations from scope rulings

*(Same posture as items 2/3's "Plan deviations" sections: each is resolved to the closest faithful alternative, flagged rather than silently resolved.)*

1. **Ruling #4's `ConsentRequest` kind `"parked-action"` — `{kind:"parked-action", actionId, decision yes|no}` doesn't fit the existing `/consent` route's contract.** Verified (`grep -rn "consentRequestSchema\|ConsentRequest\b"`): `ConsentRequest`/`consentRequestSchema` has **zero runtime call sites** anywhere in the tree today — it was shipped in item 2 as a documented type for future card rendering, never actually serialized or parsed. `ConsentResponse`/`consentResponseSchema`, by contrast, IS the real validated wire object for `/consent`, keyed by the ai SDK's `toolCallId` against an `approval-requested` part sitting inside a **thread message** (`handleConsent`'s `findApprovalPart` scans `FlowletUIMessage[]`). A parked action has neither a `toolCallId` nor a thread message — it lives in the `AutomationEngineStore`, keyed by its own store-assigned id. Forcing it through `/consent`'s `{threadId, toolCallId, toolName, response}` shape would mean inventing fake values for fields that don't apply. **Resolution:** (a) `consentRequestSchema` is widened to a discriminated union adding `kind: "parked-action"` exactly as the ruling asks — safe because nothing consumes the old single-shape schema at runtime (Task 6). (b) The actual resolve wire call gets its own small, dedicated schema (`parkedActionResolutionSchema`) and its own route (`/parked-actions/:id/resolve`), following the identical "thin adapter over a testable handler function" pattern `consent.ts`/`consent-handler.ts` already established (Tasks 7–8) — the ruling's intent (a server-validated, audited yes/no on a specific pending item) is honored; the transport shape is the one that actually fits what's being confirmed.

2. **§8's "parked action approved later re-evaluates its guard before executing" vs. the ruling's own "v1 simplification" carve-out.** The ruling explicitly pre-empts a literal reading ("re-evaluate guardExpr with the ORIGINAL scope snapshot is meaningless") and asks for "the closest honest v1." This plan defines **"trivially possible" concretely**: a step's guard (`step.if`) is re-checked fresh at resolve time ONLY when it does not reference `steps.` (i.e., it depends only on `trigger`/`run`/`user`/the enclosing loop's bindings — all of which are reconstructable at resolve time from the finalized `AutomationRun` row plus a snapshot the draft itself carries). A guard that reads `steps.<id>.output` (another step's result from the SAME run) is **never** re-checked — reconstructing a full historical `ExpressionScope` from persisted, capped `StepRecord`s is real scope creep this item doesn't take on. In that case `guardStale: true` is stamped and the frozen input executes as-is, exactly as the ruling instructs. When a guard IS re-checked and now evaluates `false` (spec's own example: "the invoice may have been paid since"), this plan resolves the action as `"approved"` (the human said yes) but **does not execute** — `resolved with skipped: true, reason: "guard no longer holds"` — the most faithful realization of the design's own worked example, not merely a documented gap (Task 5).

3. **Ruling #3's "a `resolveParkedAction(deps)` function in `@flowlet/runtime`" vs. where the exactly-once invariant actually lives.** `AutomationRunner.resume()` already solves this EXACT problem for direct-step pause approvals: its own docstring says "two rapid approvals would otherwise both pass a pre-queue check and double-execute," solved by claiming inside the per-automation `enqueue()` promise chain before executing. A bare top-level function outside the class would have to reimplement that serialization from scratch (a second, parallel source of the same race) or accept the race. **Resolution:** `resolveParkedAction` ships as a new method on `AutomationRunner` (Task 4), sibling to `resume()`, reusing `this.enqueue()`. The runtime-level "seam" the ruling wants is still there — hosts call `runner.resolveParkedAction(...)`, a portable, transport-agnostic function-shaped API — it's just attached to the object that already owns automation execution serialization instead of duplicating it.

4. **Ruling #5's "the host mounts it near the automations area — find how AutomationCard/FlowGallery surfaces mount there."** Verified: neither app has a standing "automations area" panel. `AutomationCard` is a card variant `MessageList` renders INLINE for `create_automation`/`update_automation`/etc. approvals (`isAutomationApproval`); `FlowGallery` is the saved-flowlet library shown on the Landing empty state. Automations otherwise live only inside chat turns — there is no dashboard-like surface to attach a badge to yet (that's closer to item 5's Trust screen). **Resolution:** `WaitingList` mounts as a new, small, top-level section the host places explicitly — in the accounting demo, a collapsible strip above the tab bar on `apps/demo-accounting/src/app/assistant/page.tsx` (Task 11), visible whenever the count is > 0 and collapsed to just the badge otherwise. This is a genuine, undirected product-shape choice (not fully specified by the design doc, which only shows the phone-mock Moment 8 sketch) — **flagged for Yousef's review**, not assumed as final IA.

5. **Where `ParkedAction` lives.** Not `@flowlet/core`'s frozen `Store` seam — it follows the existing precedent of `AutomationGrant`/`PendingApproval`, both of which are automation-engine-specific additive types that live in `packages/flowlet-runtime/src/automations/store.ts` on `AutomationEngineStore`, never in core. A parked action is exactly that shape of thing (engine-internal, Principal-scoped, store-assigned identity).

6. **Steps AFTER a parked step within the SAME `for_each` iteration.** The design says "the run completes everything it can; leftovers wait" but doesn't specify whether sibling steps in the same iteration, after the parked one, should still run. This plan treats a parked step like a soft-skip: the iteration's `executeSteps` continues to the next step (mirroring the existing `if`-guard-false "skipped" continuation already in the loop, Task 2). A later step that expression-references the parked step's output sees `undefined` (JSONata's normal missing-path behavior) — no crash, just an absent value the spec author is responsible for guarding against. **Flagged as a genuine judgment call, not a spec-mandated behavior** — surfaced in the report's open risks.

7. **`parkedCount`'s "additive-cheapest" placement (ruling #2).** Chosen: an incrementing integer on `AutomationRun`, bumped by the runner in the SAME `updateRun`/`finalizeRun` call that persists that batch of drafts — O(1), and it can never drift out of sync because the runner is the only writer of both the parked-action rows and this counter, in the same tick. The alternative (a live `listParkedActions(scope, {runId}).length` query on every run-list render) was rejected as a repeated store scan for a number that only ever grows monotonically within one run's lifetime.

function capParkedInput(input: unknown): { input: unknown; truncated?: boolean; bytes?: number } {
  const bytes = jsonBytes(input);
  if (bytes <= MAX_STEP_OUTPUT_BYTES) return { input };
  return { input: truncateValue(input, MAX_STEP_OUTPUT_BYTES), truncated: true, bytes };
}

```ts
  // (inside InMemoryAutomationStore)
  private parkedActions = new Map<string, ParkedAction>();
  private parkedIdCounter = 0;

  async createParkedAction(scope: Principal, input: CreateParkedActionInput): Promise<ParkedAction> {
    const id = `parked-${++this.parkedIdCounter}`;
    const { input: cappedInput, truncated, bytes } = capParkedInput(input.input);
    const action: ParkedAction = {
      id,
      tenantId: scope.tenantId,
      subject: scope.subject,
      automationId: input.automationId,
      runId: input.runId,
      stepId: input.stepId,
      tool: input.tool,
      input: cappedInput,
      ...(truncated ? { inputTruncated: true, inputBytes: bytes } : {}),
      ...(input.guardExpr !== undefined ? { guardExpr: input.guardExpr } : {}),
      ...(input.guardBindings !== undefined ? { guardBindings: input.guardBindings } : {}),
      reason: input.reason,
      tier: input.tier,
      descriptorHash: input.descriptorHash,
      requestedAt: input.requestedAt,
    };
    this.parkedActions.set(id, action);
    return action;
  }

  async listParkedActions(scope: Principal, filter: ListParkedActionsFilter): Promise<ParkedAction[]> {
    return [...this.parkedActions.values()].filter(
      (a) =>
        a.tenantId === scope.tenantId &&
        a.subject === scope.subject &&
        (filter.automationId === undefined || a.automationId === filter.automationId) &&
        (filter.runId === undefined || a.runId === filter.runId) &&
        (filter.unresolvedOnly !== true || a.resolution === undefined),
    );
  }

  async getParkedAction(scope: Principal, id: string): Promise<ParkedAction | undefined> {
    const action = this.parkedActions.get(id);
    if (!action || action.tenantId !== scope.tenantId || action.subject !== scope.subject) return undefined;
    return action;
  }

  async resolveParkedAction(
    scope: Principal,
    id: string,
    resolution: ParkedActionResolution,
    resolvedAt: string,
  ): Promise<ParkedAction> {
    const action = await this.getParkedAction(scope, id);
    if (!action) throw new Error(`parked action "${id}" not found`);
    if (action.resolution !== undefined) {
      throw new Error(`parked action "${id}" is already resolved (${action.resolution})`);
    }
    const next: ParkedAction = { ...action, resolution, resolvedAt };
    this.parkedActions.set(id, next);
    return next;
  }

  /**
   * Resolve a parked action (ENG-193 §4.6) — late, STANDALONE execution: no
   * interpreter re-run, no re-resolution of input (frozen at park time).
   * Reuses the SAME per-automation queue `resume()` relies on for its
   * "two rapid approvals must not double-execute" guarantee (spec §8's
   * parked-critical-executes-exactly-once invariant rides this, not a new
   * mechanism — plan deviation #3).
   */
  async resolveParkedAction(
    scope: Principal,
    actionId: string,
    decision: "approved" | "declined",
  ): Promise<{ ok: true; executed: boolean; skipped?: boolean; reason?: string } | { ok: false; error: string }> {
    const peek = await this.config.store.getParkedAction(scope, actionId);
    if (!peek) return { ok: false, error: `parked action "${actionId}" not found` };
    if (peek.resolution !== undefined) {
      return { ok: false, error: `parked action "${actionId}" is already resolved (${peek.resolution})` };
    }
    return this.enqueue(peek.automationId, async () => {
      // Re-peek inside the queue: a sibling concurrent call may have resolved
      // it while this one waited (the SAME race resume() guards against).
      const action = await this.config.store.getParkedAction(scope, actionId);
      if (!action || action.resolution !== undefined) {
        return { ok: false, error: `parked action "${actionId}" is already resolved` };
      }

      if (decision === "declined") {
        await this.config.store.resolveParkedAction(scope, actionId, "declined", this.now());
        return { ok: true, executed: false };
      }

      const automation = await this.config.store.get(scope, action.automationId);
      if (!automation) return { ok: false, error: `automation "${action.automationId}" not found` };
      const tools = await this.config.tools(scope, automation);
      const tool = tools[action.tool];
      if (!tool) return { ok: false, error: `tool "${action.tool}" is no longer registered` };

      // Descriptor drift (spec §8.8): a tool whose safety identity changed
      // since park time is never executed on the stale approval — the row
      // stays UNRESOLVED (re-askable), not silently declined.
      const liveHash = hashDescriptor(tool.descriptor);
      if (liveHash !== action.descriptorHash) {
        return {
          ok: false,
          error: `tool "${action.tool}" changed since this was parked — approval must be requested again`,
        };
      }

      // CLAIM before executing (the resume() pattern): a retry after this
      // point sees "already resolved" and 409s instead of double-executing.
      await this.config.store.resolveParkedAction(scope, actionId, "approved", this.now());

      // Guard re-check (deviation #2): only when it's provably self-contained
      // (no steps.* reference) — otherwise flagged stale, never re-evaluated.
      if (action.guardExpr !== undefined && !/\bsteps\s*\./.test(action.guardExpr)) {
        const run = await this.config.store.getRun(scope, action.runId);
        const scopeForGuard = {
          trigger: run?.trigger.payload,
          steps: {},
          run: { id: action.runId, automationId: action.automationId, firedAt: run?.trigger.occurredAt ?? "" },
          user: await this.claims(scope),
          ...(action.guardBindings ?? {}),
        };
        const stillHolds = await evaluateGuard(action.guardExpr, scopeForGuard);
        if (!stillHolds) {
          return { ok: true, executed: false, skipped: true, reason: "guard no longer holds" };
        }
      } else if (action.guardExpr !== undefined) {
        // References steps.* — never re-checked (deviation #2). Executes
        // with the frozen input; the WaitingList/ceremony copy is
        // responsible for saying so ("reviewed Xh after the run").
      }

      const idempotencyKey = `${action.runId}/${action.stepId}/parked-${action.id}`;
      const outcome = await tool.execute(action.input as Record<string, unknown>, { idempotencyKey });
      if (!outcome.ok) return { ok: false, error: outcome.error.message };
      return { ok: true, executed: true };
    });
  }

import { useCallback, useEffect, useRef, useState } from "react";
import { useShell } from "./context";
import type { ParkedActionRow } from "./components/WaitingList";

const POLL_MS = 30_000;

/** Polls the host's parked-action list (ENG-193 §4.6) every 30s while
 *  mounted; exposes approve/decline that re-fetch immediately after posting. */
export function useParkedActions() {
  const { parkedActions } = useShell();
  const [actions, setActions] = useState<ParkedActionRow[]>([]);
  const mounted = useRef(true);

  const refresh = useCallback(() => {
    if (!parkedActions) return;
    void parkedActions.list().then((rows) => { if (mounted.current) setActions(rows); });
  }, [parkedActions]);

  useEffect(() => {
    mounted.current = true;
    refresh();
    if (!parkedActions) return undefined;
    const id = setInterval(refresh, POLL_MS);
    return () => { mounted.current = false; clearInterval(id); };
  }, [refresh, parkedActions]);

  const approve = (actionId: string) =>
    parkedActions ? parkedActions.resolve(actionId, "yes").then(refresh) : Promise.resolve();
  const decline = (actionId: string) =>
    parkedActions ? parkedActions.resolve(actionId, "no").then(refresh) : Promise.resolve();

  return { actions, count: actions.length, approve, decline, refresh };
}
```
