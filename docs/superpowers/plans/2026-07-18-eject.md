# Wave 2: Eject Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** `npx vendo eject <surface>` — copy a shipped chrome surface's presentation source into the host repo as code the developer owns, shadcn-style, closing the customization cliff without a parts API.

**Architecture:** The published `@vendoai/ui` package ships its chrome presentation sources as eject templates (added to the package build). The CLI copies a surface's directory into the host app (default `components/vendo/<surface>/`, honoring `src/` layouts the way init does), rewrites the templates' internal relative imports so hooks/context/theme resolve from `@vendoai/ui` (pixels are copied; data/wire logic stays a package dependency), and stamps a manifest recording surface + package version for drift detection.

**Precondition:** Lane A (shelf core) MERGED — eject copies the refactored per-piece file layout. Verify the layout on main matches this plan's assumptions before building; adjust here, not there.

**Source of truth:** `docs/brainstorms/ui-usage-dx.md` (§4 customization ladder). Read it first.

**Hard boundaries:** Ejectable surfaces v1: `thread`, `activities`. (Overlay stays shipped — its `thread` injection prop is how an ejected thread mounts into it; page and slot ejection are follow-ups, not v1.) Never commit to main; open a PR from this lane's branch.

---

### Task 1: Templates in the package

**Files:** `packages/ui` build config + package manifest.

- [ ] Ship the chrome source for ejectable surfaces inside the published package (a templates path assembled at build; verify with a pack-and-inspect test the way the single-package-install work did)
- [ ] Each template set carries a generated header comment ("ejected from @vendoai/ui vX — yours to edit") and compiles standalone against `@vendoai/ui` public exports only — add a build-time check that template files import nothing package-internal that isn't publicly exported (export the few internals the thread needs, deliberately, as part of this task)

### Task 2: The eject command

**Files:** create `packages/vendo/src/cli/eject.ts` (+ test), register in `framework.ts`.

- [ ] `vendo eject thread` copies the template into the host, rewrites imports, writes the manifest (`.vendo-eject.json` in the ejected dir: surface, package version); `--list` shows surfaces; refuses to overwrite an existing dir without `--force`
- [ ] Prints the two-line swap instruction (import the local component; pass it to the overlay's `thread` prop where relevant)
- [ ] TDD path resolution (app/ vs src/app hosts), rewrite correctness, refuse/force behavior

### Task 3: Prove it end-to-end

- [ ] In demo-bank, eject the thread, make a visible presentation change (e.g. distinct bubble styling), mount it via the overlay `thread` prop and on the `/vendo` page; browser-verify + screenshots — this is the acceptance test that the cliff is actually closed
- [ ] Typecheck/build the demo with the ejected code in place; revert the demo changes before the PR unless keeping one as a living example is cleaner (worker's call; say which in the PR)

### Task 4: Drift awareness + docs

- [ ] `vendo doctor` gains a check: ejected manifests vs installed `@vendoai/ui` version — warn (never fail) on drift, pointing at the changelog
- [ ] Document the ladder (tokens → props → eject → hooks) in `docs/host-components.md`
- [ ] Full gates green; PR with screenshots; signal `needs-review` then `triage-complete` via worktree comment
