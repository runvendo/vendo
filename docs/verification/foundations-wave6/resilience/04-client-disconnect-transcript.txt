Wave 6b beat D (ENG-239) — client disconnect cancels the agent loop (Maple).
Captured 2026-07-16 against `pnpm --filter demo-bank dev` on localhost:3000.

Setup (no source changes): the host ran with
  ANTHROPIC_BASE_URL=http://127.0.0.1:8788/v1
pointed at anthropic-passthrough-proxy.mjs (`@ai-sdk/anthropic` reads that env
var), so every provider call the agent loop makes is one numbered line in the
proxy log (05-client-disconnect-proxy-log.txt). The wave-5 path under test
(ENG-238, PR #318): POST /threads hands `request.signal` to `agent.stream`
(packages/vendo/src/server.ts), which passes it to `streamText` as
`abortSignal` (packages/agent/src/agent.ts) — when it fires, the in-flight
provider call is torn down and no further step starts.

Drill: client-disconnect.mjs POSTs a real chat turn at /api/vendo/threads
(cookie-less; the wire mints an anonymous session), reads the UI-message
stream, and in abort mode disconnects the client mid-generation
(AbortController.abort() on the fetch) a fixed delay after the first
provider-generated stream part.

FINDING (flagged for follow-up, not fixed here): a REAL BROWSER closing the
tab or navigating away does NOT trigger this path under `next dev`.

The scripted aborts above and the curl kill are abrupt socket teardowns, and
Next fires `request.signal` within milliseconds. But when a real Chromium
client disconnects gracefully mid-turn, Next 16.2.9 (Turbopack, dev) ends the
POST (the request logs its duration at the disconnect moment) WITHOUT firing
`request.signal`, and the agent loop runs the whole turn against a reader
that is gone. Observed three ways, each with the turn visibly continuing in
the proxy log:

  - Orca embedded browser, tab closed 13:06:33 (POST ended at 9.3 s = the
    close): calls #16–#22 — SEVEN provider calls — issued entirely after the
    disconnect; the abandoned turn ran until 13:13:47, 7m14s past the close.
  - Plain Playwright Chromium driving the real UI (fresh profile against the
    local dev server only), page.close() at 13:21:56.6: #30–#32 issued after
    the close; abandoned turn ran 2m28s past it.
  - Same, navigating to example.com at 13:25:40.8 instead of closing:
    #36–#41 issued after the navigation; abandoned turn ran 3m33s past it.

06-client-disconnect-browser.png is the Playwright page one second before its
close: the turn mid-generation (four host tool calls done, vendo_apps_create
still input-streaming). Reproduce with browser-disconnect.mjs.

So the wave-5 abort wiring itself is correct and fast — whenever
`request.signal` fires, the loop stops within single-digit milliseconds and
issues zero further provider calls. The gap is upstream of the wire: the Next
dev server does not translate a graceful browser disconnect into a
`request.signal` abort (an abrupt client abort, it does). Not yet tested
under `next start`; whether the fix belongs in Next's disconnect detection or
in the umbrella also observing response-stream cancellation is for the
follow-up ticket. Until then, every closed-tab turn on a Next dev host runs
to completion and pays for its remaining provider calls.
