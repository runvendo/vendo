# @vendoai/ui — headless hooks + optional chrome

Status: FROZEN (wave-2 gate passed by Yousef, 2026-07-11). Changes now require a major. One job: every surface. Headless-first — every capability reachable via hooks so hosts build fully custom surfaces; shipped chrome is the fast path and adopts the host's brand so deeply Vendo is invisible. ⚑ One package absorbing the old react/client/shell/components/stage, with subpath exports. Depends on core only; React ≥ 18 peer; talks to the server exclusively over the umbrella's wire routes (09 §3).

## 1. Entry points

| Subpath | Contents |
| --- | --- |
| `@vendoai/ui` | provider, hooks, client (headless — no styles) |
| `@vendoai/ui/chrome` | the shipped surfaces (styled, theme-adopting) |
| `@vendoai/ui/tree` | the tree renderer — also consumed by served-app scaffolds (graduation requirement, 06 §2) |
| `@vendoai/ui/voice` | the voice stage driver + surface |

## 2. Client + provider

```ts
export function createVendoClient(config: { baseUrl?: string /* default "/api/vendo" */; headers?: Record<string, string> }): VendoClient;
// VendoClient: typed fetch/SSE bindings for every wire route (09 §3). Exposed for non-React consumers.

export function VendoProvider(props: {
  client?: VendoClient;
  components?: Record<string, ComponentType>;   // host catalog implementations, by name
  theme?: Partial<VendoTheme>;                  // overrides on top of server-extracted theme
  children: ReactNode;
}): JSX.Element;
```

Host components register client-side by name (matching the sync-extracted catalog descriptors); `remixable` marking happens at registration in host code and is picked up by sync (04 §1).

## 3. Hooks (the headless contract — complete list)

```ts
useVendoThread(threadId?)   → { messages, sendMessage, status, approvals: pending-in-turn, addToolApprovalResponse, stop }   // ai-SDK useChat-compatible surface
useApprovals()              → { pending: ApprovalRequest[]; decide(ids, decision) }                                       // incl. batch
useGrants()                 → { grants: PermissionGrant[]; revoke(id) }
useApps()                   → { apps: AppDocument[]; create(prompt); remove(id); fork(id) }   // remove = AppsRuntime.delete (`delete` is unusable in destructuring)
useApp(appId)           → { app, surface: OpenSurface, call(ref, args), edit(instruction), history: {list, undo}, refresh /* = re-open */ }
useAutomations()            → { automations; enable(id); disable(id); runs(filter); dryRun(id); stopRun(runId) }
useActivity()               → { events, loadMore }                       // "what has the agent done as me"
useVendoStatus()            → { posture, connected }                     // backs the no-policy notice + doctor
useVoice()                  → { state, start, stop, transcript }         // WebRTC voice stage session
useVendoTheme()             → VendoTheme                                 // resolved tokens
```

All hooks are transport-only (no styling, no portal assumptions) and SSR-safe.

## 4. Chrome (shipped surfaces — every surface type on the page)

```ts
<VendoThread />          // the conversation surface
<VendoOverlay />         // floating launcher + panel
<VendoSlot id="..." />   // inline slot: renders one of the user's apps or a pinned component in place
<VendoPage />            // full-page agent workspace (threads + apps + activity)
<VendoPalette />         // ⌘K command palette — agent reachable from anywhere
<VendoStage />           // voice
<ApprovalCard />         // the one consent surface (used in-thread and in queues)
<ActivityPanel />        // user-facing transparency
<AutomationsPanel />     // list, enable/disable, run history, dry-run, kill switch
<NoPolicyNotice />       // the loud default-posture banner (renders only when posture === "unconfigured")
```

Brand adoption: all chrome styles derive from `VendoTheme` CSS variables (colors, typography, radius, density, motion) — no hardcoded brand. Accessibility bar (contract, not aspiration): WCAG 2.1 AA, keyboard-complete, screen-reader-tested, mobile web out of the box.

## 5. The tree renderer (`@vendoai/ui/tree`)

```ts
export function TreeView(props: {
  tree: Tree;
  components: Record<string, ComponentType>;      // host catalog impls
  data?: Record<string, Json>;                    // overrides tree.data (live query results)
  onAction(req: { nodeId: string; action: string; payload?: Json }): Promise<ToolOutcome>;   // the dispatch chokepoint
}): JSX.Element;
```

- Renderers key by `formatVersion` (core §8): `TreeView` is the registered renderer for `vendo-genui/v1` — the only v0 format; an unregistered tag renders a contained notice, never breaks the page.
- Renders `vendo-genui/v1`: prewired primitives in-box; host components looked up by name; **generated components always evaluate inside the sandboxed iframe jail** (CSP `connect-src 'none'`), props in, actions out — never `eval` in the host page.
- `$path` bindings resolve against the data model; `$state` against view state (persisted per user per app via the state singleton).
- Action dispatch routes through `onAction` → the wire → guard; `fn:` refs and tool names are opaque strings here.
- Containment: an erroring node renders a contained placeholder; dangling children render skeletons (streaming); a completed tree cannot break the page.
- Pins: `<VendoSlot>` mounts approved pin code with an error boundary that falls back to the original host component — customization can degrade, the product never breaks (06 §8).
- Rung-4 cover: `<AppFrame surface={OpenSurface}>` renders the iframe (http), or the dimmed non-interactive screenshot while `resuming`.

## 6. Notices

The "no policy" default posture (05 §1) must be loud: chrome surfaces render `NoPolicyNotice` automatically; headless hosts get it via `useVendoStatus().posture` — leaving it out is a host's deliberate act.
