## 2. Client + provider

```ts
export function createVendoClient(config: { baseUrl?: string /* default "/api/vendo" */; headers?: Record<string, string> }): VendoClient;
// VendoClient: typed fetch/SSE bindings for every wire route (09 ¬ß3). Exposed for non-React consumers.

export function VendoProvider(props: {
```

Host components register client-side by name (matching the sync-extracted catalog descriptors); `remixable` marking happens at registration in host code and is picked up by sync (04 ¬ß1).

## 3. Hooks (the headless contract ‚Äî complete list)

```ts
useVendoThread(threadId?)   ‚Üí { threadId: effective ThreadId | undefined, messages, sendMessage, status, approvals: pending-in-turn, addToolApprovalResponse, stop }   // ai-SDK useChat-compatible surface
useApprovals()              ‚Üí { pending: ApprovalRequest[]; decide(ids, decision) }                                       // incl. batch
useGrants()                 ‚Üí { grants: PermissionGrant[]; revoke(id) }
useApps()                   ‚Üí { apps: AppDocument[]; create(prompt); remove(id); fork(id) }   // remove = AppsRuntime.delete (`delete` is unusable in destructuring)
useApp(appId)           ‚Üí { app, surface: OpenSurface, call(ref, args), edit(instruction), history: {list, undo}, refresh /* = re-open */ }
useAutomations()            ‚Üí { automations; enable(id); disable(id); runs(filter); dryRun(id); stopRun(runId) }
useConnections()            ‚Üí { connections: ConnectionAccount[]; refresh(); disconnect(id, connector?) }  // per-user connected accounts; ConnectionAccount is a ui-owned WIRE type mirroring 04 ¬ß3's ConnectorAccount over the umbrella routes (09 ¬ß3) ‚Äî ui still imports core only; initiate rides the connect card / client.connections
useActivity()               ‚Üí { events, loadMore }                       // "what has the agent done as me"
useThreads()                ‚Üí { threads: ThreadSummary[]; get(id); remove(id) }   // headless conversation list/get/delete
useVendoStatus()            ‚Üí { posture, connected }                     // backs the no-policy notice + doctor

// Every collection hook above also exposes the consistent, additive read shape
// `{ data, error, isLoading, refresh }` and an opt-in `pollMs` option
// (ENG-219). `data` mirrors the named field (apps/pending/grants/‚Ä¶); `error`
// carries the first fetch's failure instead of swallowing it; `refresh`
// re-fetches; `pollMs` re-fetches on a cadence with no remount. `useApps` also
// exposes `exportApp(id)`/`importApp(bytes)`.
useVoice()                  ‚Üí { state, start, stop, transcript }         // WebRTC voice stage session
useVendoTheme()             ‚Üí VendoTheme                                 // resolved tokens
```

All hooks are transport-only (no styling, no portal assumptions) and SSR-safe.

> **Amendment (2026-07-14, ENG-229 Phase 1):** the voice contract grows additively for a recoverable, observable session. `VoiceState` / `VoiceSessionState` add `"reconnecting"`; `VoiceSessionHandle` adds optional `setMuted(muted)`; the driver event union adds `{ type: "amplitude"; level: number }`; and `useVoice()` additionally returns `error: { message: string } | null`, `muted`, `setMuted(muted)`, and `amplitude`. `RealtimeVoiceDriverOptions` adds optional `connectTimeoutMs` (20 seconds by default). The realtime driver preserves transcript across bounded internal reconnects, reapplies mute to replacement microphone streams, and reports microphone permission, session mint, SDP exchange, timeout, and exhausted-reconnect failures through the existing error event.

> **Amendment (2026-07-14, ENG-229 Phase 2):** the voice seam adds a session-view event `{ type: "view"; view: { id: string; appId: string; payload: UIPayload } }`; `useVoice()` additionally returns `views`, deduplicated by `id` with same-id events replacing the stored view. `VendoStage` adds optional `onSessionEnd()`, called after its exit settle beat. The shipped stage polls `client.approvals.pending()` only while a voice session is active, decides through `client.approvals.decide`, auto-yields its transcript drawer to pending consent, and routes rendered view actions through `client.apps.call`. The realtime driver does not yet originate view events; that producer-side integration remains additive future work.

> **Amendment (2026-07-14, ENG-319, voice live pipeline):** the voice driver seam grows additively for a live tool-calling session. The driver event union already carries `{ type: "view"; view: VoiceSessionView }`; ENG-319 adds the producer side: `RealtimeVoiceDriverOptions` gains optional `act: VoiceToolBridge` (`{ tools: Array<Record<string, unknown>>; onToolCall(call: VoiceToolCall, session: VoiceActSession): Promise<unknown> }`), whose `tools` ride the provider `session.update` and whose `onToolCall` handles every realtime function call ‚Äî its resolved value is returned to the model as the function output and a fresh response is requested. `VoiceActSession.emitView` originates driver view events. The shipped bridge `createVoiceActBridge({ client, threadId? })` exposes ONE `vendo_act` function tool that runs a REAL guarded agent turn over `POST /threads` per call (03 ¬ß4): minted views stream into the stage feed, parked guard approvals reach the stage consent bar (ENG-229), and the turn resumes via the existing assistant-upsert approval-response path (AGENT-12) with the guard authoritative over execution (05 ¬ß1). No new server surface and no wire change ‚Äî the live voice agent rides the same pipeline the chat surface does.

## 4. Chrome (shipped surfaces ‚Äî every surface type on the page)

```ts
<VendoThread />          // the conversation surface
<VendoOverlay />         // floating launcher + panel
<VendoSlot id="..." />   // inline slot: renders one of the user's apps or a pinned component in place
<VendoPage />            // full-page agent workspace (threads + apps + activity)
<VendoPalette />         // ‚åòK command palette ‚Äî agent reachable from anywhere
<VendoStage />           // voice
<ApprovalCard />         // the one consent surface (used in-thread and in queues); renders the invalidated-grant notice when the request carries invalidatedGrant (01 ¬ß5, 05 ¬ß2)
<ConnectCard />          // inline connect-required surface (04 ¬ß3.1): initiate ‚Üí OAuth popup ‚Üí poll active ‚Üí automatic retry of the original call
<ConnectedAccountsPanel /> // persistent connected-accounts list + disconnect, in VendoPage chrome settings
<ActivityPanel />        // user-facing transparency
<AutomationsPanel />     // list, enable/disable, run history, dry-run, kill switch
<NoPolicyNotice />       // the loud default-posture banner (renders only when posture === "unconfigured")
```

Brand adoption: all chrome styles derive from `VendoTheme` CSS variables (colors, typography, radius, density, motion) ‚Äî no hardcoded brand. Accessibility bar (contract, not aspiration): WCAG 2.1 AA, keyboard-complete, screen-reader-tested, mobile web out of the box.

## 5. The tree renderer (`@vendoai/ui/tree`)

```ts
export function TreeView(props: {
  tree: Tree;
  components: Record<string, ComponentType>;      // host catalog impls
  data?: Record<string, Json>;                    // overrides tree.data (live query results)
  onAction(req: { nodeId: string; action: string; payload?: Json }): Promise<ToolOutcome>;   // the dispatch chokepoint
}): JSX.Element;
```

- Renderers key by `formatVersion` (core ¬ß8): `TreeView` is the registered renderer for `vendo-genui/v1` ‚Äî the only v0 format; an unregistered tag renders a contained notice, never breaks the page.
- Renders `vendo-genui/v1`: prewired primitives in-box; host components looked up by name; **generated components always evaluate inside the sandboxed iframe jail** (CSP `connect-src 'none'`), props in, actions out ‚Äî never `eval` in the host page.
- `$path` bindings resolve against the data model; `$state` against view state (persisted per user per app via the state singleton).
- Action dispatch routes through `onAction` ‚Üí the wire ‚Üí guard; `fn:` refs and tool names are opaque strings here.
- Containment: an erroring node renders a contained placeholder; dangling children render skeletons (streaming); a completed tree cannot break the page.
- Pins: `<VendoSlot>` mounts approved pin code with an error boundary that falls back to the original host component ‚Äî customization can degrade, the product never breaks (06 ¬ß8).
- Rung-4 cover: `<AppFrame surface={OpenSurface}>` renders the iframe (http), or the dimmed non-interactive screenshot while `resuming`.

## 6. Notices

The "no policy" default posture (05 ¬ß1) must be loud: chrome surfaces render `NoPolicyNotice` automatically; headless hosts get it via `useVendoStatus().posture` ‚Äî leaving it out is a host's deliberate act.

## Amendments

### 2026-07-14 ‚Äî Effective thread identity surfaced

- **Changed:** Added the effective `threadId` to `useVendoThread`'s return value. A supplied id is exposed while it is validated, a stale id transitions to `undefined`, and a subsequent server-confirmed or server-minted id is exposed after a turn.
- **Why:** Headless hosts need to observe stale-thread recovery and persist the effective server identity without issuing a noisy missing-thread detail request.
- **Approved by:** Yousef, 2026-07-14.

### 2026-07-15 ‚Äî In-client venue mount (06-apps ¬ß9) reconciled with ¬ß5

- **Changed:** ¬ß5's "generated components always evaluate inside the sandboxed iframe jail ‚Ä¶ never `eval` in the host page" describes the DEFAULT venue. 06-apps ¬ß9 (frozen in the same wave) defines the host-approved exception: an `InClientApproval` pinning the exact version content hash lets that version's generated components mount natively in the host page. The renderer now honors the additive, server-authoritative `payload.inClient` verdict: `granted: true` (written only by the apps runtime's hash-pin verification) mounts in-page through a closed module space (React + captured sub-sources only); every other state ‚Äî missing approval, `version-changed`, or any mount failure ‚Äî stays in / drops back to the jail, the version-changed drop-back with a loud in-surface notice. In-thread view parts are never the approved venue.
- **Why:** ENG-288 M4 ships the OSS enforcement half of the locked in-client split (apps-block design spec, 2026-07-14, decisions 1‚Äì2): approval record format, hash-pin verification, host-page mount, drop-back-to-iframe. Cloud owns the review console that mints approvals.
- **Authorized by:** the Yousef-approved apps-block design spec (`docs/superpowers/specs/2026-07-14-apps-block-design.md`, locked decisions 1‚Äì2).

### 2026-07-15 ‚Äî Connected accounts + loud invalidation surfaces (ENG-261/262, parent ENG-264)

- **Changed:** ¬ß3 adds `useConnections()`; the typed client gains `client.connections` bindings for the umbrella's connection routes (09 ¬ß3). The hook's `ConnectionAccount` is a ui-owned wire type mirroring 04 ¬ß3's `ConnectorAccount` ‚Äî ui keeps its core-only dependency rule (00 overview); the shape crosses as JSON over the wire, never as an actions import.
- **Changed:** ¬ß4 adds `<ConnectCard />` (in-flow connect-required ‚Üí OAuth ‚Üí automatic retry) and `<ConnectedAccountsPanel />`; `<ApprovalCard />` renders the invalidated-grant notice. All brand-native per the existing theme rules.
- **Why:** Per-user connected accounts and loud grant invalidation need first-class surfaces in both the headless and chrome tiers (ENG-262/261, landed).
- **Authorized by:** the Yousef-approved block-actions design spec (`docs/superpowers/specs/2026-07-14-block-actions-design.md`).

### 2026-07-15 ‚Äî Consistent hook read shape + polling + headless parity (ENG-219)

- **Changed (additive):** every ¬ß3 collection hook (`useApps`, `useApprovals`, `useGrants`, `useActivity`, `useConnections`, `useAutomations`, plus the single-resource `useApp`) now also returns `{ data, error, isLoading, refresh }`. `data` mirrors the existing named field (`apps`/`pending`/`grants`/‚Ä¶, all retained for back-compat); `error` carries the initial fetch's failure instead of swallowing it into a silent `undefined`; `isLoading` reflects the first load; `refresh` re-fetches. Collection hooks accept an opt-in `{ pollMs }` option that re-fetches on a cadence with no remount (a newly-pending approval now appears on its own).
- **Changed (additive):** ¬ß3 adds `useThreads()` ‚Üí `{ threads, get(id), remove(id) }` (+ the shared read shape) so the conversation list/get/delete is reachable headlessly rather than only via `client.threads`. `useApps` adds `exportApp(id)`/`importApp(bytes)`, exposing the previously client-only app transfer over a hook.
- **Why:** headless consumers could not tell empty / failed / loading apart, nothing refreshed without a remount, and threads + app export/import had no headless coverage (block-ui design spec, "headless parity in scope").
- **Authorized by:** the Yousef-approved block-ui design spec (`docs/superpowers/specs/2026-07-14-block-ui-design.md`).

### 2026-07-16 ‚Äî Orgs client surface removed

- **Changed:** `useOrgs()`, `<OrgsPanel />`, `client.orgs`, and the org wire types are removed from this block. Orgs are a Vendo-Cloud-side feature; the OSS repo keeps only the `cloud-required` seam.
- **Why:** simplify-v2 kill-list A5 ‚Äî orgs were implemented against the host's local store, contradicting the 2026-07-16 data-residency decision (Cloud enabled = data stored with Vendo).
- **Authorized by:** the Yousef-approved simplify-v2 kill-list (`docs/superpowers/specs/2026-07-16-simplify-v2-kill-list-design.md`, ¬ßA5).
